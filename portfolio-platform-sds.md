## 1. Introduction

### 1.1 Purpose
This document specifies the design of the Portfolio Platform: a self-managed content system in which the portfolio (home, projects, about, contact) is the first content served by a general-purpose publishing engine, not a fixed set of pages. It defines the architecture, content model, component responsibilities, data flow, and quality requirements needed to build it.

### 1.2 Scope
Covers the CMS admin, the content API, the public-facing Next.js application, and the contract between them. Excludes hosting account setup and DNS configuration, which are operational tasks outside the design.

### 1.3 Definitions
- **Page** — a generic, CMS-authored document that renders at a URL.
- **Block** — a single unit of content (hero, rich text, gallery, etc.) that composes a Page.
- **Content API** — the headless CMS's data layer, treated as the system's single source of truth.
- **Revalidation** — the on-demand refresh of a statically generated route, triggered by a publish event rather than a scheduled rebuild.

### 1.4 Design goals, in priority order
1. Publishing a new page requires no code change and no deploy.
2. Content and rendering logic are strictly separated.
3. The system extends by adding block types, not by adding routes.
4. The site is fast by default (static) and fresh on demand (revalidated).
5. The end product reads as deliberately made — not assembled from defaults — without adding operational fragility.

## 2. System Overview

### 2.1 Context
The platform has three collaborating parts and one human actor:

```
┌───────────────┐        ┌──────────────────┐        ┌─────────────────┐
│   CMS Admin   │───────▶│   Content API     │───────▶│   Next.js App    │───────▶ Visitor
│ (editor UI)   │        │ (CMS + Postgres)  │        │ (static + ISR)   │
└───────────────┘        └──────────────────┘        └─────────────────┘
                                    │                          ▲
                                    └──── webhook: revalidate ─┘
```

- **CMS Admin** — where the site owner creates and edits Pages, Projects, and Navigation.
- **Content API** — stores content as structured records and exposes it over REST/GraphQL. This is the only source of truth; nothing about *what* the site says is allowed to live outside it.
- **Next.js App** — fetches from the Content API and renders it. Statically generated by default; individual routes are revalidated on publish via webhook.

### 2.2 Why this shape
The API boundary is what turns a website into a platform: the same content could later feed an RSS feed, a mobile client, or a CLI without any change to how content is authored. This is a design property to protect, not an aspiration to defer.

### 2.3 Out of scope for v1
Multi-author permissions, localization, and a public comments system. The content model should not actively block these later, but none are built now.

## 3. Architecture Decisions

| Layer | Decision | Rationale |
|---|---|---|
| Frontend framework | Next.js, App Router | Static generation plus on-demand revalidation in one framework; large ecosystem. |
| CMS | Payload CMS, self-hosted | Content types defined in code (version-controlled schema); admin UI generated from that schema; no vendor lock-in. |
| CMS alternative | Sanity | Use if self-hosting is undesirable — faster to start, hosted admin, less schema control. |
| Database | PostgreSQL | Payload's preferred adapter; cheap, standard, easy to back up. |
| Media storage | S3-compatible bucket (e.g. Cloudflare R2) | Decouples images from the app server; CDN-friendly. |
| Styling | Tailwind CSS | Fast enough to leave time for a distinctive visual system rather than a utility grid alone. |
| Frontend hosting | Vercel | Native ISR and webhook support, zero-config CDN. |
| CMS hosting | Railway / Render / Fly.io | Simple always-on host for admin + database. |
| Auth | Built into Payload | Single admin user; a separate identity system is unjustified complexity. |

**Decision rule for future additions:** use a proven off-the-shelf solution for anything that is not the reader-facing experience (auth, storage, admin plumbing). Reserve custom engineering for the block system, the content model, and the design layer described in Section 7.

## 4. Content Model

This is the core of the design. Every later capability — new sections, new page types — should be achievable by extending this model, not by branching around it.

### 4.1 `Page`
Generic and reusable for Home, About, Contact, or any page created later.

```
Page {
  title            string
  slug             string          // "/", "about", "contact", "now", ...
  seo              { metaTitle, metaDescription, ogImage }
  showInNav        boolean
  navOrder         number
  blocks           Block[]         // ordered, composable
}
```

### 4.2 `Block` (discriminated union)
The mechanism that makes pages buildable rather than fixed.

```
Block =
  | HeroBlock        { heading, subheading, image, ctaLabel, ctaLink }
  | RichTextBlock    { content }
  | ProjectGridBlock { heading, filterByTag?, limit? }
  | ImageGalleryBlock{ images: Image[] }
  | TimelineBlock    { items: { year, title, description }[] }
  | ContactFormBlock { heading, fields }
  | CustomEmbedBlock { html }        // escape hatch for the unforeseen
```

Adding a new block type (e.g. Testimonials) means: define its shape in Payload, write one React component to render it, register it in the block renderer. No route changes, no data migration of existing pages.

### 4.3 `Project`
A specialized content type — structurally a Page with sharper edges, not a separate system.

```
Project {
  title            string
  slug             string
  coverImage       Image
  summary          string          // shown in grid/cards
  techStack        Tag[]
  liveUrl?         string
  repoUrl?         string
  gallery          Image[]
  body             Block[]         // same block system as Page
  featured         boolean
  publishedDate    date
}
```

### 4.4 `Navigation` (singleton)
```
Navigation { items: NavItem[] }

NavItem {
  label            string
  linkType         "page" | "project" | "external"
  target           Page | Project | url
  children?        NavItem[]       // dropdown support
}
```
**Design decision:** start by auto-populating this list from every `Page` with `showInNav: true`, sorted by `navOrder`, for zero-maintenance navigation. Introduce manual `NavItem` overrides only when dropdowns or non-Page links are needed.

### 4.5 `SiteSettings` (singleton)
```
SiteSettings {
  siteTitle, tagline
  logo
  socialLinks       { platform, url }[]
  defaultSeoImage
  footerText
}
```

### 4.6 Entity relationships
- `Page.blocks` and `Project.body` both reference the same `Block` union — one renderer serves both.
- `Navigation` optionally derives from `Page` records where `showInNav = true`.
- `SiteSettings` is referenced globally (header, footer, default SEO) but never referenced by a Page or Project.

## 5. Component Design

### 5.1 Routing
```
/app
  /(site)
    /page.tsx                  → home (fetches the Page with slug "/")
    /[slug]/page.tsx           → catch-all for every CMS-defined Page
    /projects/page.tsx         → project index (grid of all Projects)
    /projects/[slug]/page.tsx  → individual project detail page
  /(cms)                       → only if Payload is embedded in the same app;
                                  otherwise a separate service/repo
```

`[slug]/page.tsx` is deliberately ignorant of content: it fetches the `Page` document for the requested slug, iterates `blocks`, and delegates rendering — it must never contain page-specific logic.

### 5.2 `BlockRenderer`
The single most important component in the system — the mechanism that lets adding a page mean adding data, not adding code.

```tsx
function BlockRenderer({ block }) {
  switch (block.type) {
    case "hero":         return <Hero {...block} />;
    case "richText":      return <RichText {...block} />;
    case "projectGrid":   return <ProjectGrid {...block} />;
    case "imageGallery":  return <ImageGallery {...block} />;
    case "timeline":      return <Timeline {...block} />;
    case "contactForm":   return <ContactForm {...block} />;
    default:               return null;
  }
}
```

**Design constraint:** each block component owns its own layout and styling entirely. No block component may assume anything about which page it appears on or what surrounds it.

### 5.3 Publish-to-live sequence
1. Editor creates or edits a `Page` in the CMS admin (adds blocks, sets `showInNav`).
2. Editor publishes.
3. Payload fires a webhook to `/api/revalidate` with the affected slug(s).
4. The route handler validates a shared secret, then calls `revalidatePath()` for the affected page and, if navigation changed, the layout.
5. The next request to that path serves the freshly generated version — no rebuild, no redeploy.

## 6. Data Flow & Rendering Strategy

| Concern | Approach |
|---|---|
| Default rendering | Static generation via `generateStaticParams`, covering all known slugs at build time. |
| Freshness | On-demand revalidation via webhook — content goes live in seconds, not on a rebuild schedule. |
| Draft content | Next.js draft mode exposes unpublished content at a private preview URL. |
| SEO metadata | `generateMetadata()` per Page/Project, sourced directly from CMS fields — no metadata maintained by hand. |
| Sitemap | `/sitemap.xml` generated at build time from the full list of published slugs. |

## 7. Visual & Experience Design Direction

The architecture in Sections 2–6 is what makes new pages possible without a deploy. This section is what determines whether the *result* feels made by hand rather than assembled from a template — it is binding on implementation, not optional polish.

### 7.1 Design principle
Treat every block component as a piece of typography and layout craft, not a generic content placeholder. A `RichTextBlock` and a `HeroBlock` on a templated site look interchangeable in structure; here, each block's internal spacing, type scale, and rhythm should be considered individually rather than inherited from a single global "card" style.

### 7.2 Typography
Specify (not default to) a distinct pairing:
- A **display face** with real character, used only for headings and the hero — not the same face used for everything, and not a generic system sans.
- A **body face** chosen for long-form reading comfort, since the `RichTextBlock` is where writing lives.
- A **utility/mono face**, if used, reserved for metadata: dates, tags, captions — never for prose.

Avoid the reflexive combination of a warm cream background with a high-contrast serif and a terracotta accent — this has become a generic "AI-generated" default and will read as templated regardless of execution quality. Choose a palette and pairing derived from the portfolio owner's actual work and sensibility, not from what is fastest to reach for.

### 7.3 Color and identity
Define the palette as 4–6 named values in `SiteSettings` or a design-tokens file, not scattered inline. The palette should be specific enough that swapping it would visibly change the site's personality — a sign it was chosen, not defaulted to.

### 7.4 The block system as a design system
Because every page is assembled from blocks, visual consistency depends entirely on the discipline of the block components, not on page-level templates. Each block must:
- Define its own vertical rhythm (consistent internal spacing scale).
- Respect a shared type scale and color token set, without reimplementing either.
- Degrade gracefully with partial data (e.g. a `HeroBlock` with no image should not leave a visibly broken gap).

### 7.5 Motion
Motion should be deliberate and rare rather than ambient. Prefer one considered moment (e.g. a page-load reveal on the hero, or a subtle scroll transition between blocks) over per-element hover effects scattered throughout. If in doubt, remove the animation.

### 7.6 Content voice
Written content (labels, empty states, form errors, button text) is part of the design, not filler. Buttons should name the action in the visitor's terms ("Send message," not "Submit"); empty states (e.g. an empty `ProjectGridBlock`) should read as intentional, not broken.

### 7.7 Acceptance check
Before a block type is considered done, it should pass: does this look like it was designed for *this* portfolio, or would it look identical dropped into any other one? If the latter, the block needs another pass.

## 8. Non-Functional Requirements

| Requirement | Specification |
|---|---|
| **Performance** | Images served via Next.js `<Image>` and the CDN bucket; static generation keeps per-request server work near zero. |
| **SEO** | Every Page/Project carries its own meta fields; sitemap generated from published slugs. |
| **Security** | Only the CMS admin requires auth; the public site is read-only against the API; contact form is rate-limited server-side. |
| **Extensibility** | The Content API must remain consumable by non-Next.js clients (mobile app, RSS, CLI) without schema changes. |
| **Availability** | Static pages remain servable even if the CMS/database is briefly unavailable, since pages are pre-rendered. |

## 9. Testing Strategy

| Layer | What is verified | Method |
|---|---|---|
| `BlockRenderer` | Every block type renders without crashing on valid data | Unit tests (Vitest/Jest), one per block type, with snapshots |
| Content fetching | API failures degrade gracefully (missing page → 404, not a crash) | Unit tests with mocked API responses |
| Routing | `[slug]/page.tsx` resolves known slugs and 404s on unknown ones | Integration tests against a seeded test database |
| Contact form | Validation rejects malformed input; valid submissions succeed | Integration test against the actual form-handling route |
| Critical paths | Home loads, project grid links to the correct detail pages, nav lists all published pages | End-to-end tests (Playwright) — a small, targeted set |
| Payload schema | Field validation (required fields, slug uniqueness) behaves as defined | Payload's own hook/validation tests |

Coverage is prioritized toward the block renderer and routing layer, since those are what "add a page yourself" depends on not silently breaking. Full coverage is not a goal for a single-owner site.

## 10. Security Design

- **CMS admin**: strong password policy or magic-link/OAuth login if supported; admin route on its own subdomain, not linked from the public site.
- **Webhook endpoint** (`/api/revalidate`): requires a shared secret validated server-side; without it, the URL alone would let anyone trigger a rebuild.
- **Contact form**: server-side rate limiting (e.g. 5 submissions/hour/IP), a honeypot field or CAPTCHA, and sanitized/escaped input before storage or email.
- **Secrets**: database URL, API keys, and webhook secret live in environment variables on the host — never in the repository or client bundle.
- **CORS**: the Content API accepts requests only from the known frontend origin(s), never `*`.
- **Dependency hygiene**: automated update checks (Dependabot or similar) given how quickly Payload and Next.js evolve.
- **Backups**: automated daily Postgres backups — the database, not the code, is where the content lives.

## 11. Phased Build Plan

| Phase | Deliverable |
|---|---|
| **1 — Skeleton** | Next.js app against a hardcoded content shape matching Section 4. Home, Projects index, Project detail, About, Contact, with real design applied per Section 7. |
| **2 — Wire up the CMS** | Payload + Postgres stood up; content types defined exactly as modeled; hardcoded content migrated in; frontend pointed at the live API. |
| **3 — Full dynamic system** | `[slug]/page.tsx` catch-all and `BlockRenderer` implemented; `Navigation` singleton driving the real nav; webhook revalidation route live; draft preview enabled. |
| **4 — Polish** | Remaining block types added; motion per Section 7.5; admin field descriptions and validation refined; sitemap and structured data added. |

---

## 12. Summary

Every visible element is generated from a database record. Every editable element lives in an admin panel. The only thing that ever requires a redeploy is the *capability* to render a new kind of block — never a new page. The architecture in Sections 2–6 guarantees that mechanically; Section 7 is what determines whether visitors can feel the difference between a system that publishes content and one that was made with care.
